## Case Sensitivity Findings

### Question Answered
**Is the `site_catalog` ACF field case sensitive?**

**Answer**: ✅ **YES, it IS case sensitive**

### Current Live Configuration ✅
```bash
# Verified on live server:
site_catalog = 'Vimergy_catalog'  ← CORRECT format ✅
               'Test_catalog'      ← CORRECT format ✅
```

**Live site is already using the correct format!**

### Required Format
**ACF Field Values**:
- ✅ `Vimergy_catalog` (Capital first letter, rest lowercase)
- ✅ `Gaia_catalog` (Capital first letter, rest lowercase)
- ✅ `HP_catalog` (Exception: all caps acronym)
- ✅ `DCG_catalog` (Exception: all caps acronym)

**WordPress Roles**:
- ✅ `access-vimergy-user` (all lowercase)
- ✅ `access-gaia-user` (all lowercase)

### Why It Matters
The plugin converts roles to catalog names using `ucfirst()`:
- Role: `access-vimergy-user` → Catalog: `Vimergy_catalog`

Then uses strict comparison (`in_array($cat, $user_catalogs, true)`):
- `'Vimergy_catalog' === 'Vimergy_catalog'` → Match ✅
- `'Vimergy_catalog' === 'vimergy_catalog'` → No match ❌

## Important Notes

### Case Sensitivity ⚠️

**When adding new catalogs, ALWAYS use this format:**
- ACF: `{Brand}_catalog` (Capital first letter, rest lowercase)
- Role: `access-{brand}-user` (All lowercase)

**Examples**:
- ✅ `Gaia_catalog` + `access-gaia-user`
- ❌ `gaia_catalog` + `access-gaia-user` (won't work!)
- ❌ `GAIA_CATALOG` + `access-gaia-user` (won't work!)

### Current Configuration
Your live site is **already using the correct format**:
- ✅ `Vimergy_catalog`
- ✅ `Test_catalog`
- ✅ `HP_catalog`
- ✅ `DCG_catalog`

**Just maintain this format for any new catalogs you add.**
