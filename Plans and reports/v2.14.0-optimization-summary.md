## The Core Discovery: Hook Priority Strategy

### What We Learned

Third-party plugins often hook into WordPress at very high priorities (900000+) and set specific query parameters. Instead of trying to hook early and modify queries, we can:

1. **Research** the plugin to find their hook priority
2. **Hook at priority +1** (right after them)
3. **Filter their results array** using fast array operations
4. **Use cached data** for blazing performance

### Why This Matters

**Before**:
- Complex query modifications
- Fighting plugin architecture
- Performance issues
- Unreliable results

**After**:
- Simple array filtering (`array_diff`)
- Working with plugin architecture
- Fast (uses cached data)
- Reliable results

### The Pattern

```php
// Step 1: Find their hook priority
// FiboSearch: add_action( 'pre_get_posts', ..., 900001 );

// Step 2: Hook +1 higher
add_action( 'pre_get_posts', 'our_filter', 900002 );

// Step 3: Filter their array
function our_filter( $query ) {
    $their_array = $query->get( 'post__in' );
    $blocked = get_cached_blocked_products();
    $filtered = array_diff( $their_array, $blocked );
    $query->set( 'post__in', $filtered );
}
```

**Method**:

**Why It's Fast**:
1. Cached blocked products list (no DB queries)
2. Simple array operation (`array_diff`)
3. Runs once per search query
4. FiboSearch's heavy lifting already done (search, score, sort)

## Architecture Insights

### Our Multi-Layered Approach

**Layer 1: General Queries** (Priority 5)
- Hook: `pre_get_posts` at priority 5
- Method: `post__not_in` exclusion
- Catches: Shop, archives, most product queries
- Fast: Uses cached list

**Layer 2: FiboSearch Specific** (Priority 900002)
- Hook: `pre_get_posts` at priority 900002 (AFTER FiboSearch)
- Method: Filter `post__in` array
- Catches: FiboSearch search results page
- Fast: Array operation on already-searched products

**Layer 3: WooCommerce Universal** (Default priority)
- Hook: `woocommerce_product_data_store_cpt_get_products_query`
- Method: Modify query args to exclude restricted products
- Catches: `wc_get_products()` calls (sliders, widgets, etc.)
- Fast: Modifies args before query runs

**Layer 4: Client-Side Fallback** (JavaScript)
- Hook: DOM manipulation with jQuery
- Method: Hide elements after render
- Catches: FiboSearch dropdown (SHORTINIT mode)
- Only option: SHORTINIT bypasses all PHP

**Why This Works**: Each layer catches what the previous layers miss, creating comprehensive coverage.

## Key Takeaways for Future Development

### 1. Research Before Implementing
Don't assume hooks work how you think. Check the plugin's actual code:
```bash
# Find their hooks
grep -n "add_action\|add_filter" PLUGIN_FILE.php

# Find what they set
grep -n "query->set" PLUGIN_FILE.php
```

### 2. Hook Priority Matters
Standard WordPress hooks (priority 10) may not be enough. Third-party plugins use 900000+ priorities. Hook at their priority +1.

### 3. Filter Results, Not Queries
Letting plugins do their work then filtering results is:
- Faster (they've done the heavy lifting)
- More reliable (respects their caching)
- Easier to maintain (less complex)

### 4. Always Use Cached Data
```php
// GOOD ✅
$blocked = pam_get_blocked_products_cached(); // 30min cache

// BAD ❌  
$blocked = expensive_query_every_time();
```

### 5. Document What Doesn't Work
We spent hours discovering that `dgwt/wcas/tnt/search_results/products` doesn't fire. Document these failures to prevent repeating them.
